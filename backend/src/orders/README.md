# RFM Analysis Documentation

This document explains how the **Recency, Frequency, Monetary (RFM)** model is implemented in `rfm_analysis.js` to segment customers based on their purchasing behavior. 

The analysis runs in a 3-step pipeline:

## Step 1: Raw Data Aggregation (MongoDB)
The system uses a MongoDB Aggregation pipeline to calculate the raw behavioral data for every customer in the database via a two-stage query:

### Stage 1: `$group` 
Gathers every single order in the database and bundles them by the person who bought them.

```javascript
$group: {
    _id: "$userId",                        // Groups the orders by customer ID
    monetary: { $sum: "$totalPrice" },     // M: Adds up the totalPrice of every lifetime order
    frequency: { $count: {} },             // F: Counts the exact number of orders placed
    lastOrderDate: { $max: "$createdAt" }  // Finds the most recent date they placed an order
}
```

### Stage 2: `$project`
Cleans up the grouped data and mathematically converts the `lastOrderDate` timestamp into a simple integer of "Days Ago".

```javascript
$project: {
    userId: "$_id",   
    monetary: 1,      // '1' simply passes the calculated monetary value through unchanged
    frequency: 1,     // '1' simply passes the calculated frequency count through unchanged
    
    // R: Calculates exactly how many days ago their 'lastOrderDate' was
    recencyDays: {
        $floor: {     // Rounds down to a whole number of days
            $divide: [
                // Subtracts the last order date from "today" (returns milliseconds)
                { $subtract: [today, "$lastOrderDate"] },
                // Divides by the number of milliseconds in a single day
                1000 * 60 * 60 * 24 
            ]
        }
    }
}
```

## Step 2: Scoring via Quintiles (1-5 Scale)
Raw numbers aren't completely useful on their own, so we score them relatively from 1 to 5. The system ranks all users and splits them into five equal 20% groups (quintiles).

This is done directly in JavaScript using the `assignQuintileScore` helper function:

```javascript
const assignQuintileScore = (data, field, descending = true) => {
    // 1. Sort the users to create a leaderboard
    const sorted = [...data].sort((a, b) => 
        descending ? b[field] - a[field] : a[field] - b[field]
    );
    
    // 2. Loop through every user to calculate their personal score
    return data.map(item => {
        // Find their exact position (0 is 1st place, n-1 is last place)
        const rank = sorted.findIndex(s => s.userId === item.userId);
        
        // 3. Divide rank by total users to place them in a 20% quintile bracket
        // Math: 5 - roundedDown((rank * 5) / totalUsrs)
        let score = 5 - Math.floor((rank * 5) / data.length);
        
        // Safety bounds
        if (score < 1) score = 1;
        if (score > 5) score = 5;
        
        // Attach the new score to their profile (e.g. monetaryScore: 5)
        return { ...item, [`${field}Score`]: score };
    });
};
```

### Breakdown of the logic:
1. **Sorting (`descending` flag):** It sorts all customers from best to worst. 
   - *Monetary & Frequency:* Highest numbers are put at the top of the leaderboard (`descending: true`).
   - *Recency:* Lowest numbers (fewest days ago) are put at the top of the leaderboard (`descending: false`).
2. **Ranking:** The code uses `.findIndex` to figure out what mathematical rank the customer holds on the newly sorted leaderboard.
3. **Chunking into 5:** It takes their rank, multiplies it by 5, and divides it by the total user count. The top 20% of users get a score of `5`, the next block gets a `4`, down to the lowest 20% who get a `1`.

*Example Result:* A customer receives an object like `{ rfmCode: "525" }` meaning they ordered very recently, ordered an average amount, but spent a massive amount.

## Step 3: Segment Mapping
Finally, the 3-digit score generated by Step 2 is routed through the `getPutlerSegment()` JavaScript function to give the user a human-readable marketing label. 

The function uses a "waterfall" logic structure, checking high-value profiles first and falling back to lower-value profiles. A user falls into the very first `if` statement bracket they mathematically qualify for:

```javascript
const getPutlerSegment = (R, F, M) => {
    // 1. Champions: Bought recently, buy often and spend the most
    if (R >= 4 && F >= 4 && M >= 4) return "Champions";
    
    // 2. Loyal Customers: Buy on a regular basis. Responsive to promotions.
    if (R >= 2 && F >= 3 && M >= 3) return "Loyal Customers";
    
    // 3. Can't Lose Them: Used to purchase frequently but haven't returned for a long time.
    if (R <= 2 && F >= 4 && M >= 4) return "Can't Lose Them";
    
    // 4. At Risk: Purchased often but a long time ago.
    if (R <= 2 && F >= 3) return "At Risk";
    
    // 5. New Customers: Bought most recently, but only once/few times.
    if (R >= 4 && F <= 1) return "New Customers";
    
    // 6. Promising: Bought recently, but haven't spent much.
    if (R === 3 && F <= 1) return "Promising";
    
    // 7. Potential Loyalist: Recent customers, average frequency.
    if (R >= 3 && F >= 2 && F <= 3) return "Potential Loyalist";
    
    // 8. Needs Attention: Average recency and frequency, maybe didn't buy very recently.
    if (R >= 2 && R <= 3 && F >= 2 && F <= 3) return "Needs Attention";
    
    // 9. About to Sleep: Below average recency and frequency.
    if (R === 2 && F <= 2) return "About to Sleep";
    
    // 10. Lost: Lowest scores across the board.
    if (R <= 1 && F <= 1 && M <= 1) return "Lost";
    
    // 11. Hibernating: Last purchase was long back, low frequency.
    if (R <= 2 && F <= 2) return "Hibernating";
    
    // Fallbacks just in case a combination drops through the main checks
    if (R >= 4) return "Potential Loyalist";
    if (R === 3) return "Needs Attention";
    return "Lost";
};
```

### Breakdown of the logic:
- The segment thresholds are checked sequentially. If someone is a perfect `555`, they are caught immediately by the very first condition (`Champions`) and the function stops.
- If someone is a `333`, they skip condition 1 and 2, and are caught by condition 7 (`Potential Loyalist`). 
- This structure prioritizes **Top Tier** customers first, moves down to **Win-Backs** (high value but low recency), checks **Recent** but low-frequency buyers next, and finally cascades all the way down to **Slipping / Lost**.
- **The Fallback:** If an absolute mathematical anomaly occurs that doesn't explicitly match one of the 11 nested conditions, the function includes a safety-net at the bottom. Anyone with decent Recency (`R >= 3`) is saved from mistakenly becoming `"Lost"`, while those who haven't ordered in a long time default correctly to `"Lost"`.
